# 提示词、术语表、润色与软件设置教程

这篇教程不是要求你一次把所有功能都用上，而是帮你理解：什么时候该写提示词，什么时候该填术语表，什么时候需要润色，以及项目设置怎样配得比较稳。

如果你是第一次使用 AiNiee-Next，建议先按 [图文快速上手教程](README_QUICK_START.md) 跑通一次。等确认 API、路径和模型都能正常工作后，再回来看这篇教程调整翻译质量。

新手建议优先使用 CLI/TUI，也就是黑色控制台里的菜单界面。不要因为它看起来像“黑框框”就害怕；它其实是最线性的入口，会一步一步问你要做什么，先选 API，再选项目设置，再选提示词，最后开始翻译。WebUI 看起来更直观，但第一次打开时反而可能不知道应该先点哪里。WebUI 更适合已经理解基本流程后，用来远程查看任务、管理队列，或者在局域网内不方便直接操作主机时使用。

## 1. 先分清四类内容

很多翻译效果不好，并不是模型不行，而是把不同类型的要求全塞进提示词里，导致模型不知道什么最重要。

在 AiNiee-Next 里，建议这样分工：

- **提示词**：告诉模型“你要怎么翻”，例如文风、语气、格式要求、是否保留标点和占位符。
- **术语表**：告诉模型“这些词必须怎么翻”，例如人名、地名、组织名、技能名、专有设定。
- **世界观/角色/写作风格**：告诉模型“这部作品是什么样的”，例如时代背景、人物性格、说话方式。
- **润色**：在翻译完成后，让模型把译文改得更自然、更像中文，但不改变原意。

简单说：提示词管规则，术语表管固定译名，角色和世界观管上下文，润色管最终读感。

## 2. CLI、WebUI 和 MCP 应该怎么选

AiNiee-Next 有多个入口。它们不是互相替代的关系，而是面向不同使用场景。

### 新手优先用 CLI/TUI

CLI/TUI 就是启动 `Launch.bat` 或 `uv run ainiee_cli.py` 后看到的菜单界面。它看起来像控制台，但实际操作并不复杂。

新手建议优先使用 CLI/TUI，原因很简单：

- 流程是线性的：主菜单会告诉你每一步能做什么。
- 选项集中：API 配置、项目设置、提示词、翻译、润色都在菜单里。
- 出错时提示更直接：验证 API、路径错误、模型错误通常会在终端里直接显示。
- 不需要先理解网页布局：跟着编号输入即可。

不要把控制台的黑色背景理解成“程序很难”。对第一次配置来说，它反而比网页更容易，因为你不需要猜按钮在哪，也不需要先理解整个界面。

### WebUI 适合什么人

WebUI 是网页控制面板，优点是直观、适合看状态，也适合局域网访问。但对完全新手来说，它可能会有一个问题：刚打开时你能看到很多页面和按钮，却不知道第一步应该从哪里开始。

所以建议：

- 第一次配置 API、提示词、项目路径时，优先走 CLI/TUI。
- 已经跑通过一次翻译后，再打开 WebUI 做监控和管理。

WebUI 比较适合这些场景：

- 翻译任务在家里或宿舍的电脑上跑，你想用另一台电脑、手机或平板查看进度。
- 人在公司、学校、图书馆等场所，不方便直接操作运行任务的电脑。
- 主机放在服务器、NAS、远程桌面、宿舍电脑上，只需要在局域网内查看状态。
- 需要拖拽调整任务队列，或更直观地查看 RPM、TPM、成功率、错误率。
- 想在浏览器里管理 Profile、插件、队列或缓存。

WebUI 不是“更高级所以必须用”。它更像远程控制面板。能在本机前正常操作时，新手先用 CLI/TUI 更稳。

### MCP 适合什么人

MCP 是给 LLM 客户端使用的接口，不是普通用户必须配置的功能。你可以把它理解成：让支持 MCP 的 AI 客户端通过工具调用 AiNiee-Next，而不是让人手动点菜单或网页。

MCP 适合这些场景：

- 你使用 Codex、Claude Desktop、Cherry Studio 或其他支持 MCP 的客户端。
- 你希望 AI 助手帮你读取配置、管理队列、检查任务状态、操作部分 WebServer 能力。
- 你在做自动化工作流，希望 LLM 客户端能以受控方式调用 AiNiee-Next。

如果你只是想翻译小说、漫画、字幕或游戏文本，不需要先配置 MCP。MCP 是进阶功能，可以等基本翻译流程熟悉后再研究。

## 3. 如何正确编写提示词

提示词不需要写得很长。好的提示词应该清楚、稳定、可执行，而不是堆很多抽象形容词。

建议包含这几类要求：

1. **身份和任务**：告诉模型它是在做文学翻译、游戏文本翻译、字幕翻译，还是台词润色。
2. **语言方向**：例如从日文翻译成简体中文，或从英文翻译成简体中文。
3. **格式要求**：逐行翻译，不合并行，不删除序号、占位符、变量、换行符、标签。
4. **文风要求**：自然、口语、文学化、轻小说风、严肃、古风等，选一两个核心方向即可。
5. **禁止事项**：不要总结，不要解释，不要自行删减，不要把占位符翻译掉。

一个比较稳的翻译提示词可以这样写：

```text
你是一名专业文学翻译。请将{source_language}文本翻译成自然流畅的{target_language}。

请逐行翻译，不要合并行，不要拆分行。
请保留原文中的序号、标记符、换行符、转义符、代码变量和占位符。
花括号 {...}、方括号 [...]、百分号变量、HTML 标签、游戏脚本控制符等内容，如果明显不是普通文本，请尽量保持原样。

翻译时优先保证意思准确，其次保证中文自然。
人物台词要符合说话人的语气，不要把所有角色都翻成同一种腔调。
不要总结，不要解释，不要添加原文没有的信息。

请只输出译文。
```

如果翻译的是小说，可以加一句：

```text
叙述部分保持文学感，台词部分保持自然口语，不要过度书面化。
```

如果翻译的是游戏脚本，可以加一句：

```text
优先保持脚本格式稳定，任何疑似变量、命令、标签、控制符都不要翻译。
```

如果翻译的是字幕，可以加一句：

```text
译文要简洁，适合屏幕阅读，不要写得过长。
```

不建议这样写提示词：

```text
请翻译得信达雅，要优美，要有感觉，要符合原作，要像大师一样，要完全正确。
```

这种要求看起来很高，但对模型来说不够具体。它不知道应该优先保证格式、准确性、人物语气，还是文字华丽程度。

## 4. 提示词不要承担术语表的工作

如果你希望“月読”永远翻译成“月读”，“羽衣プロトコル”永远翻译成“羽衣协议”，不要把这些词一条条写进提示词正文里。应该放进术语表。

提示词里只需要写规则，例如：

```text
请严格遵守术语表。遇到术语表中的原文词条时，优先使用术语表给出的译名。
```

术语越多，越应该使用术语表，而不是继续加长提示词。提示词太长会增加成本，也更容易让模型忽略真正重要的格式要求。

## 5. 如何正确填写术语表

术语表适合放“必须统一译名”的内容：

- 人名、昵称、称号
- 地名、国家、组织、学校、公司
- 技能、道具、系统名、专有名词
- 反复出现并且容易被模型翻乱的词
- 有特殊含义的普通词

术语表不适合放每一个普通单词。普通词交给模型按上下文翻译即可。

一个好的术语表条目应该包含三部分：

- `原文`：源语言里的写法。
- `译文`：你希望固定使用的译名。
- `备注`：这个词是什么，什么时候使用，有没有特殊语气或设定。

示例：

| 原文 | 译文 | 备注 |
| --- | --- | --- |
| 月読 | 月读 | 全球最大的虚拟空间，不要翻成“读月”。 |
| 羽衣プロトコル | 羽衣协议 | 允许虚拟体获得真实触觉反馈的同步协议。 |
| かぐや | 辉夜 | 女主角，语气天真、任性，有公主感。 |
| 月の使者 | 月之使者 | 系统派出的清理程序，不是普通神话使者。 |

备注不是越长越好。备注的作用是帮模型理解这个词，不是把整段设定都塞进去。

比较好的备注：

```text
女主角，语气天真、任性，有公主感。
```

不太好的备注：

```text
这是一个非常重要的人物，一定一定要翻译好，她贯穿全文，性格很复杂，后面还有很多剧情，千万不要出错。
```

后者情绪很多，但有效信息少。

## 6. 术语表的优先级和维护方式

刚开始翻译一部作品时，不需要先整理几百条术语。建议先做一个“小而准”的术语表：

1. 先放主要人物名。
2. 再放核心地名、组织名、能力名。
3. 翻译过程中发现模型反复翻错的词，再补进去。
4. 每次补完术语后，可以重新跑后续内容；已经翻过的内容如果不满意，再按需要重翻或润色。

术语表最怕的是冲突。例如同一个原文一会儿写“辉夜”，一会儿写“辉耀”；或者两个不同原文都填了同一个译名但备注又互相矛盾。发现译名混乱时，先检查术语表，而不是马上改提示词。

如果一个词在不同上下文里确实要翻成不同意思，不要简单写死。可以在备注里说明：

```text
原文：Master
译文：主人
备注：角色称呼对方时使用“主人”；如果是系统职位或技能等级，请按上下文翻译。
```

## 7. 世界观、角色和写作风格怎么写

如果作品比较长，只有提示词和术语表通常不够。模型还需要知道作品背景和人物关系。

世界观适合写：

- 故事发生在什么时代、地点、社会结构下。
- 重要组织、技术、魔法体系、游戏系统是什么。
- 哪些词有特殊设定含义。

角色信息适合写：

- 原名和译名。
- 性别、年龄、身份。
- 性格。
- 说话方式。
- 和其他角色的关系。

写作风格适合写：

- 叙述是轻松、严肃、压抑、华丽，还是冷静。
- 台词偏口语还是书面。
- 是否保留吐槽、粗口、敬语、古风表达。

示例：

```text
本作是近未来科幻恋爱题材。现实世界冷清压抑，虚拟世界华丽热闹。
翻译时叙述部分保持轻小说式的流畅感，避免过度文言。
女主角辉夜说话天真、任性，偶尔带有公主般的高傲。
管理员八千代语气优雅、平静，有距离感，不要翻得太活泼。
```

这些内容应该写得像给译者看的资料，而不是像给模型下命令。资料越清楚，模型越容易保持一致。

## 8. 什么时候需要润色

润色不是必须步骤。它适合在以下情况使用：

- 译文意思基本正确，但读起来不自然。
- 长篇小说翻译后，想统一中文文风。
- 台词太机械，人物声音不明显。
- 原文已经是中文，只想改得更流畅。

润色不适合用来修复严重错译。如果译文把意思翻反了、人名翻错了、变量弄丢了，应该先修正翻译设置、提示词或术语表，再重新翻译。不要指望润色自动救回所有错误。

## 9. 润色模式怎么选

AiNiee-Next 的润色可以理解为两种常见场景：

- **润色译文**：先翻译外文，再把翻译结果改得更像自然中文。这是最常用的模式。
- **润色原文**：原文已经是中文，不需要翻译，只想进行中文改写和润色时使用。

如果你是在翻译日文、英文、韩文小说，通常选择润色译文。流程是：

1. 先完成翻译。
2. 检查术语和格式没有大问题。
3. 再执行润色。
4. 最后导出或人工检查。

润色提示词可以这样写：

```text
你是一名中文文学编辑。请在不改变原意、不增删剧情信息的前提下，润色下面的译文。

请让中文更自然、流畅，适当调整语序。
请保持人物语气，不要把所有台词改成同一种风格。
请保留原有的序号、占位符、标签、变量和换行结构。
不要解释，不要总结，只输出润色后的文本。
```

如果你希望润色力度轻一点，可以加：

```text
只修正生硬、拗口和明显不自然的表达，不要大幅改写句子。
```

如果你希望润色力度强一点，可以加：

```text
可以在不改变原意的前提下重组语序，让句子更符合中文小说表达。
```

不要让润色提示词同时要求“完全不改字”和“大幅变得优美”。这两个目标本身冲突。

## 10. 软件设置怎么配比较合理

新用户建议先追求稳定，而不是一开始就追求极限速度。

推荐起步设置：

| 设置项 | 建议值 | 说明 |
| --- | --- | --- |
| 源语言 | `auto` 或明确语言 | 不确定时用 `auto`，知道原文语言时可以明确填写。 |
| 目标语言 | `Chinese` | 翻译成中文时保持这个即可。 |
| 每次翻译行数 | `10` 到 `30` | 小说可从 `20` 开始；台词短可稍高；长段落可稍低。 |
| 参考上文行数 | `2` 到 `5` | 一般 `3` 就够。太大更贵，也可能引入干扰。 |
| 请求超时 | `60` 到 `120` 秒 | 网络不稳或模型较慢时调高。 |
| 线程数/并发 | 先从 `5` 到 `10` 开始 | 确认稳定后再逐步提高。 |
| 思考模式 | 默认关闭或按模型建议开启 | 普通翻译不一定需要；复杂长句或推理模型可测试开启。 |
| Token 模式 | 新手先不启用 | 先用行数模式更直观。 |

如果你使用 DeepSeek 这类速度较快、额度较宽的在线 API，可以在稳定后逐步提高线程数，例如从 `10` 试到 `20`、`30`、`50`。如果出现大量 `429`、超时、连接失败或成功率下降，就把线程数降回来。

速度不是只看线程数。线程数过高时，可能会触发限流、网络拥塞或本机连接数问题，最终反而更慢。比较可靠的做法是看运行界面的成功率、错误率、RPM、TPM 和预计耗时。

## 11. 行数、上下文和 Token 怎么理解

“每次翻译行数”决定一次请求里塞多少行文本。

行数太小：

- 上下文不足，人物称呼和语气容易断。
- 请求次数变多，速度和成本可能变差。

行数太大：

- 模型更容易漏行、串行、格式错乱。
- 单次失败损失更大。
- 长文本可能超过模型上下文限制。

一般小说建议从 `20` 行左右开始。字幕、游戏短句可以适当增加；含大量长段落、复杂格式或代码变量的文本可以降低到 `10` 到 `15`。

“参考上文行数”是给模型一点前文，帮助它理解当前句子。通常 `3` 行比较稳。不要为了“上下文越多越好”直接拉到很高，因为模型可能被过多旧内容干扰，也会增加 Token 消耗。

Token 模式适合更熟悉模型上下文限制的用户。新用户优先使用行数模式，排查问题更简单。

## 12. 思考模式要不要开

思考模式不是越开越好。它适合复杂推理、长难句、设定密集文本，可能提升理解力，但也可能增加耗时和费用。

建议：

- 普通轻小说、游戏台词：先不开，效果不好再试。
- 设定复杂、长句多、隐喻多：可以测试开启。
- 批量大规模翻译：先用少量章节 A/B 测试，再决定是否全量开启。
- 如果开启后速度明显下降、费用变高但质量提升不明显，就关闭。

不同 API 对思考参数支持不完全一致。如果验证 API 或翻译时报参数错误，先关闭思考模式再试。

## 13. 一套稳妥的完整流程

如果你不知道从哪里开始，可以按这个流程做：

1. 先申请 API Key，验证 API 能正常使用。
2. 用默认提示词翻译一小段，不要一上来翻完整本。
3. 检查结果里最明显的问题：人名是否统一，格式是否保留，语气是否自然。
4. 把人名、地名、核心设定加入术语表。
5. 根据作品类型微调提示词，例如小说重视文风，游戏脚本重视格式。
6. 再翻译一小段，对比效果。
7. 效果稳定后再翻译整本。
8. 翻译完成后，如果中文生硬，再执行润色。
9. 最后抽查关键章节、人物名、特殊格式和输出文件。

这比一次写一个很长的提示词更可靠。翻译质量通常是靠“小步测试、逐步修正”调出来的。

## 14. 常见问题

**人名总是不统一怎么办？**

先检查术语表，不要只改提示词。把主要人物名放进术语表，并开启术语表功能。

**译文漏掉变量或标签怎么办？**

在提示词里明确要求保留变量、占位符、标签、控制符；同时降低每次翻译行数，减少格式压力。

**翻译很慢怎么办？**

先确认不是 API 本身慢。再逐步提高线程数。提高后如果错误率上升，就说明已经超过当前网络或账号限额的稳定范围。

**润色后意思变了怎么办？**

降低润色力度，在润色提示词里写“只修正不自然表达，不要大幅改写，不要增删信息”。如果仍然变动太大，说明这个模型不适合强润色，或者需要人工检查重点章节。

**提示词写得越长越好吗？**

不是。提示词越长，成本越高，也越容易让模型忽略关键要求。把固定译名放术语表，把背景放世界观，把人物语气放角色信息，提示词本身保持清楚即可。

**需要每本书都单独建一套配置吗？**

如果只是临时测试，可以共用配置。如果是长篇作品，建议单独建 Profile，避免不同作品的术语表、角色设定和提示词互相污染。

## 15. 最推荐的新手配置

第一次正式翻译长篇作品时，可以用这套保守配置：

- 选择在线 API 预设并验证成功。
- 每次翻译行数：`20`
- 参考上文行数：`3`
- 请求超时：`60` 或 `120`
- 线程数：先 `5` 到 `10`，稳定后再提高
- 术语表：开启，只放主要人物和核心设定
- 思考模式：先关闭
- 润色：翻译完成后再决定是否执行

如果这套配置能稳定完成，再根据速度和质量逐步调整。稳定的低错误率，比一开始把并发拉满更重要。

## 16. 进阶功能应该按什么顺序学

AiNiee-Next 的功能很多，但不建议一次全部学完。比较合理的顺序是：

1. **先学会 CLI/TUI 基础流程**：启动、配置 API、设置路径、选择提示词、开始翻译。
2. **再学提示词和术语表**：解决翻译质量和译名统一问题。
3. **再学 Profile**：给不同作品拆分配置，避免互相污染。
4. **再学润色**：在译文基本正确后改善中文读感。
5. **再学任务队列**：批量跑多个文件。
6. **再学 WebUI**：远程查看、队列管理、可视化监控。
7. **最后再学 MCP**：让 LLM 客户端接入 AiNiee-Next，做自动化或辅助管理。

不要一上来就研究 MCP、队列、高并发、缓存、插件。新手最常见的问题不是“功能不够”，而是 API、路径、术语表、提示词还没配稳，就直接开始大规模任务。先把最小流程跑通，后面的功能才有意义。

## 17. Profile 的进阶用法

Profile 是非常重要但容易被忽略的功能。它可以把不同作品、不同任务策略隔离开。

建议至少准备这些 Profile：

| Profile 类型 | 用途 |
| --- | --- |
| `Default` | 临时测试，不放重要长期术语。 |
| `小说-作品名` | 某部长篇小说专用，保存它的术语表、角色、世界观、提示词。 |
| `游戏-作品名` | 某个游戏脚本专用，重点保护变量、标签和脚本格式。 |
| `字幕-通用` | 字幕翻译专用，提示词强调简洁和屏幕阅读。 |
| `精修-作品名` | 翻译完成后用于润色、检查、二次处理。 |

如果你长期翻译不同作品，Profile 能省很多麻烦。否则你会遇到这种情况：上一部作品的术语表里有“月读”“羽衣协议”，下一部奇幻作品里模型还在被这些术语影响，结果翻译风格和译名都变奇怪。

一个实用习惯是：正式翻译前先复制一个干净 Profile，再为当前作品添加术语、角色和世界观。

## 18. 任务队列的进阶用法

任务队列适合批量执行，但前提是单个任务已经验证过。

适合放进队列的任务：

- 同一部小说的多个分卷。
- 同一个游戏项目的多个文本文件。
- 同一套配置下的多个字幕文件。
- 需要半夜挂机顺序执行的一批任务。

不适合直接放进队列的任务：

- API 还没验证成功。
- 提示词和术语表还没测试过。
- 文件格式你还不确定能不能正确导出。
- 每个文件需要完全不同的语言、模型、提示词或术语表。

使用队列前，建议先做一次“小样本检查”：

1. 单独翻译一个短文件。
2. 检查输出是否正常打开。
3. 检查人名、变量、标签、章节结构有没有问题。
4. 确认没问题后，再把同类文件加入队列。

队列适合扩大已经稳定的流程，不适合替代测试流程。

## 19. WebUI 的详细使用建议

WebUI 的优势是直观、远程、适合监控。但它不一定适合第一次上手。

第一次用 AiNiee-Next 时，建议你先用 CLI/TUI。原因不是 WebUI 不好，而是 CLI/TUI 的操作是线性的：它会把你带到 API 配置、项目设置、提示词选择、任务启动这些步骤里。WebUI 虽然有页面、有按钮、有图表，但新手刚打开时可能会不知道先点哪里。

WebUI 更适合这些情况：

- 任务已经开始跑，你想看进度。
- 想看 RPM、TPM、成功率、错误率。
- 想在局域网内用手机、平板或另一台电脑查看任务。
- 运行任务的电脑不在手边，比如放在宿舍、家里、服务器、NAS 或远程主机上。
- 人在公司、学校、图书馆等地方，不方便直接操作运行任务的电脑。
- 想用浏览器管理队列、Profile 或插件。

一个比较自然的用法是：

1. 在运行任务的电脑上用 CLI/TUI 配好 API、路径和提示词。
2. 先用 CLI/TUI 启动或验证任务。
3. 需要远程查看时，再启动 WebServer。
4. 用浏览器打开 WebUI，看进度或调整队列。

如果你只是坐在电脑前第一次配置软件，不需要强迫自己用 WebUI。CLI/TUI 的黑色界面只是外观，不代表它难用。它反而能减少“我打开网页后不知道该点哪里”的困惑。

使用 WebUI 时请注意：

- 不建议暴露到公网。
- 只在可信局域网内访问。
- 不要在公共屏幕上展示 API Key 或敏感配置。
- 如果任务正在运行，不要频繁改关键配置。
- 如果 WebUI 页面里的功能和 CLI/TUI 不完全一致，优先以 CLI/TUI 的主流程为准。

## 20. MCP 的详细使用建议

MCP 是给 AI 客户端使用的，不是给普通用户手动点的界面。

你可以这样理解：

- CLI/TUI：人用键盘操作。
- WebUI：人用浏览器操作。
- MCP：LLM 客户端通过工具调用来操作。

如果你只是要翻译文件，不需要 MCP。如果你想让 Codex、Claude Desktop、Cherry Studio 或其他支持 MCP 的客户端帮你查看配置、管理队列、上传文件、读取任务状态，这时才需要 MCP。

MCP 常见接入方式有两种：

| 方式 | 适合场景 |
| --- | --- |
| `stdio` | 本机 LLM 客户端自动启动 AiNiee MCP，适合 Codex 这类工具。 |
| `streamable-http` | AiNiee 先启动一个 MCP HTTP 服务，客户端通过 URL 连接。 |

如果你不知道该选哪个：

- 本机工具优先看 `stdio`。
- 需要固定连接地址时看 `streamable-http`。
- 普通翻译用户可以先完全不管 MCP。

MCP 的重点不是“开一个端口给 AI 随便访问”，而是“通过受控工具暴露能力”。所以安全规则必须遵守：

- LLM 客户端只能通过 MCP 工具调用 AiNiee-Next。
- 不要让 LLM 绕过 MCP，直接访问 WebUI、localhost 或局域网 HTTP 接口。
- MCP 读取到 API Key 等敏感字段时会脱敏。
- 脱敏占位符不是可用密钥，不能当真实 Key 保存回配置。
- 修改 `mcp_server_port`、`mcp_server_host` 等高级设置前，要确认自己知道影响。

首次接入 MCP 后，建议让客户端先调用：

```text
get_mcp_usage_manual
get_mcp_security_policy
get_mcp_tool_categories
get_mcp_tool_catalog(category="需要的分类")
get_mcp_validation_checklist
```

这几个工具会告诉 AI 客户端当前 MCP 有哪些能力、参数怎么写、哪些接口受限，以及为什么不能绕过 MCP 直连 WebUI。端点目录默认按分类读取，避免一次性把全部 Web API 端点塞进上下文。

## 21. 插件和高级功能怎么取舍

插件、RAG、翻译检查、缓存、自动化这些功能都很有用，但它们适合在基础流程稳定后再启用。

建议优先级：

1. **术语表**：最先学，直接影响译名一致性。
2. **角色和世界观**：长篇作品很有用，能提升语气和设定一致性。
3. **润色**：译文正确但生硬时使用。
4. **翻译检查**：用于发现漏译、错译、格式异常。
5. **RAG 或上下文增强**：长篇作品、前后文依赖强时再考虑。
6. **上下文缓存**：提示词和设定很长、平台支持时再开启。
7. **MCP/自动化**：需要接入外部 AI 客户端或工作流时再配置。

如果启用某个高级功能后错误变多，先关闭它，回到基础流程确认问题。排查问题时功能越少越容易定位。

## 22. 不同项目的推荐配置

### 小说翻译

小说最重要的是语气、术语和上下文。

推荐：

- 每次翻译行数：`15` 到 `25`
- 参考上文行数：`3`
- 术语表：开启
- 角色信息：建议填写主要角色
- 世界观：长篇或设定复杂时填写
- 润色：翻译完成后按需要执行
- 线程数：先 `5` 到 `10`，稳定后提高

提示词重点：

```text
叙述部分保持自然文学感，台词符合人物语气。
不要随意删减、概括或解释原文。
请保留格式、序号、占位符和换行结构。
```

### 游戏脚本

游戏脚本最重要的是格式安全。

推荐：

- 每次翻译行数：`10` 到 `20`
- 参考上文行数：`2` 到 `3`
- 术语表：开启
- 线程数：不要一开始太高
- 润色：谨慎使用，避免改坏变量和标签

提示词重点：

```text
只翻译玩家可见文本。
保留变量、标签、控制符、角色字段、文件路径和脚本结构。
不要翻译代码、命令、键名和占位符。
```

### 字幕翻译

字幕最重要的是简洁和可读。

推荐：

- 每次翻译行数：`20` 到 `40`
- 参考上文行数：`2`
- 译文长度：尽量短
- 润色：可用于压缩生硬长句

提示词重点：

```text
译文要简洁，适合屏幕阅读。
不要添加解释，不要写得过长。
保留时间轴和字幕结构。
```

### 漫画翻译

漫画对白最重要的是短、自然、适合气泡。

推荐：

- 术语表：开启，尤其是人名、招式名、组织名。
- 角色语气：建议填写。
- 润色：可用于让对白更自然，但不要过度扩写。
- WebUI：适合查看任务状态或后续人工精修。

提示词重点：

```text
对白要短，适合漫画气泡。
语气自然，不要过度书面化。
拟声词可按中文习惯处理，但不要改变画面含义。
```

## 23. 常见误区

**误区一：WebUI 一定比 CLI/TUI 更适合新手。**

不一定。WebUI 更直观，但不是更线性。新手第一次配置时，CLI/TUI 的菜单编号反而更清楚。

**误区二：并发越高越好。**

并发过高会带来限流、超时和错误率上升。真正重要的是稳定吞吐，不是线程数字。

**误区三：提示词越长越专业。**

提示词越长，越贵，也越容易互相冲突。能放术语表的不要放提示词，能放角色信息的不要塞进系统提示词。

**误区四：润色能修复所有翻译问题。**

润色只能改善表达。错译、漏译、术语错、格式错应该先回到翻译阶段修。

**误区五：MCP 是普通用户必须配置的功能。**

不是。MCP 是给 LLM 客户端和自动化流程用的。只想翻译文件时，可以完全不用。

**误区六：第一次就应该把完整本书丢进去。**

不建议。先翻一小段，确认 API、提示词、术语表、输出格式都没问题，再全量跑。

## 24. 推荐的进阶路线

如果你已经完成第一次翻译，可以按这个路线继续提升：

1. 为当前作品新建独立 Profile。
2. 整理主要人物、地名、组织名、能力名到术语表。
3. 给主要角色补一句说话风格。
4. 给作品补一小段世界观说明。
5. 用一小段文本测试提示词。
6. 根据结果只改一个方向：术语、格式、语气或自然度。
7. 稳定后提高线程数。
8. 需要批量时再使用任务队列。
9. 需要远程监控时再使用 WebUI。
10. 需要 AI 客户端自动操作时再配置 MCP。

这条路线的核心是：先稳定，再提速；先小样本，再批量；先手动理解流程，再交给 WebUI、队列或 MCP 扩展。
